# 配置与测试结构优化总结

## 1. 本次优化目标

本次工作主要解决两个问题：

1. Spring Boot 项目配置入口过多、职责重复，部分配置仍带有旧版本兼容代码。
2. 自动化测试、在线调试脚本和源码契约测试混放，部分测试文件重复代码较多，`DetectionTaskServiceImplTest` 体积过大。

优化遵循“先建立约束测试，再修改生产配置，最后执行全量回归”的原则，没有为了减少文件数量而删除安全、租户、数据库迁移、Kafka 幂等和核心业务测试。

## 2. 配置优化结果

| 优化项 | 优化前 | 优化后 |
| --- | --- | --- |
| Spring Boot 主配置 | `application.yml` 与 `application.properties` 并存 | 统一为 `application.yml` |
| MyBatis 依赖 | 原生 MyBatis Starter 与旧 MyBatis-Plus Starter 并存 | 仅使用 `mybatis-plus-spring-boot3-starter` |
| MyBatis 配置 | 多个配置类、自动配置排除和空 Runner 兼容代码 | 使用官方自动配置，只保留 Mapper 扫描和分页插件 |
| `.env` 优先级 | 通过 `addFirst` 覆盖系统环境变量 | 作为低优先级本地默认值，系统环境变量和启动参数优先 |
| 数据库初始化 | 存在重复字符集设置和引用不存在 SQL 的初始化类 | 使用 Hikari `connection-init-sql`，保留唯一业务种子入口 |
| Web 编码 | 自定义过滤器、消息转换器与 Spring Boot 配置重复 | 统一使用 Spring Boot 编码配置 |
| 日志 | 默认生成滚动日志文件 | 默认仅输出到控制台，日志级别由环境变量控制 |
| 测试依赖 | 显式引入未使用的 H2 和重复 Mockito | 删除 H2，Mockito 由 `spring-boot-starter-test` 提供 |
| 环境模板 | 本地与 Docker 配置职责不够清晰 | `.env.example` 用于本地，`deploy/docker.env.example` 用于容器部署 |

配置拓扑契约会持续检查：

- 项目只能存在一个主应用配置文件；
- 只能使用 Spring Boot 3 的 MyBatis-Plus Starter；
- 不得重新加入 Bean 覆盖、SQL Runner 等历史兼容开关；
- 不得恢复已删除的重复配置类；
- 不得重新启用默认文件日志。

## 3. 测试结构优化结果

### 3.1 Python 测试与人工诊断分层

需要已启动后端、前端或浏览器的脚本已移动到 `scripts/diagnostics`：

- `agent_live_smoke.py`
- `login_browser_debug.py`
- `login_browser_debug_alt.py`
- `usage_status_browser_debug.py`

这些脚本只允许人工触发，不再被默认 pytest 收集。`tests_python` 只保留离线、确定性的自动化测试。

### 3.2 前端契约测试收敛

前端测试新增统一源码读取助手：

```text
frontend/tests/helpers/project-source.cjs
```

原有 24 个单主题契约文件按职责合并为 5 个领域文件：

```text
assistant-contracts.test.cjs
quality-contracts.test.cjs
detection-contracts.test.cjs
deployment-contracts.test.cjs
security-shell-contracts.test.cjs
```

加上助手自身测试，前端测试根目录由 25 个测试文件收敛为 6 个。原有测试名称和断言完整保留。源码文件缺失时助手会直接抛出 `ENOENT`，避免否定断言因为读取到空字符串而产生假通过。

### 3.3 检测任务测试拆分

原 `DetectionTaskServiceImplTest` 超过 1200 行，共 38 项测试。现在通过共享 Fixture 拆分为：

| 测试类 | 测试数 | 职责 |
| --- | ---: | --- |
| `DetectionTaskUploadTest` | 13 | 创建任务、上传确认、重试 |
| `DetectionTaskEventTest` | 7 | Kafka 完成事件、重复事件、后台回调 |
| `DetectionTaskQualityTest` | 13 | 质检流转、复核、处置、返工 |
| `DetectionTaskTraceTest` | 5 | 批次、工单、缺陷与质量追溯 |

`DetectionTaskServiceFixture` 统一管理 6 个 Mock、管理员安全上下文、Service 构造和 Mockito 严格会话。每项测试结束时执行 `finishMocking()` 并清理 `SecurityContextHolder`，避免未使用 Stub 被忽略或权限上下文在线程间泄漏。

## 5. 为什么使用了较多后台智能体

本次共创建了 **32 个后台智能体角色**，加上主智能体共 33 个执行角色。这里的“32 个”是角色实例数量，不代表 32 个智能体一直同时运行，也不代表进行了 32 次独立功能开发。

具体构成为：

| 类型 | 数量 | 主要职责 |
| --- | ---: | --- |
| 实现智能体 | 10 | 分别处理配置拓扑、MyBatis、YAML、日志、依赖、Python、前端和 Java 测试拆分 |
| 阶段规格审查智能体 | 10 | 核对实现是否完整满足每个阶段的要求 |
| 阶段质量审查智能体 | 10 | 检查正确性、测试假阳性、状态泄漏和维护性问题 |
| 最终综合审查智能体 | 2 | 对全部配置与测试重构做最终规格和质量检查 |
| 合计 | 32 | 不包含主智能体 |

使用较多后台智能体的主要原因如下：

1. **采用了分阶段实现与双重审查流程。** 每个较大的阶段由实现智能体完成，再分别进行规格审查和代码质量审查。
2. **减少自我审查偏差。** 实现者容易默认自己的方案正确，独立审查更容易发现遗漏。本次实际发现了应用名丢失、缺失源码导致测试假通过、Mockito 严格校验丢失、安全上下文未清理、README 命令过期等问题。
3. **控制单次改动范围。** MyBatis、YAML、日志、Python、前端和 Java 测试拆分相互独立，分开处理更容易定位失败和回滚。
4. **执行了多轮修复复审。** 审查发现问题后，会让原实现智能体修复，再由审查智能体复核；复审属于同一角色的后续工作，不会重复创建新角色。
5. **用户选择了后台智能体驱动的执行方案。** 因此本次优先保证可验证性和审查独立性，而不是用单一智能体一次性完成全部修改。

后台并发槽位最多为 4，其中包含主智能体，因此通常只有 1 个实现智能体或 1 个审查智能体在后台工作，并非 32 个智能体同时消耗资源。

## 6. Token 消耗说明

### 6.1 为什么无法给出精确 Token 数

当前会话提供给智能体的工具只显示任务状态、代码变更和命令输出，**没有提供主智能体及各后台智能体的逐角色 Token 使用统计**。本次也没有在开始时创建带 Token 预算和用量跟踪的 Goal，因此无法从现有工具中得到可信的总 Token 数。

因此，本总结不提供猜测值。使用对话字数、提交数量或智能体数量估算 Token 都会产生较大误差，不能作为实际计费依据。

精确用量应以调用端保存的 API `usage` 字段、产品用量页面或完整会话导出中的 Token 统计为准，并将主智能体和所有后台智能体的输入、输出以及可能存在的缓存或推理 Token 汇总。

### 6.2 本次 Token 消耗较高的原因

- 项目上下文和历史需求较长；
- 后台智能体需要读取任务上下文和相关代码；
- Maven、pytest、Node 和前端构建输出较多；
- 每个阶段执行了实现、规格审查、质量审查和修复复审；
- 最终又进行了跨模块全量审查与全量验证。

### 6.3 后续如何降低 Token 消耗

在风险允许的情况下，可以采用以下方式：

1. 将多个低风险机械任务交给同一个实现智能体；
2. 只在阶段里程碑执行双重审查，而不是每个小任务都审查；
3. 给后台智能体传递最小必要上下文，不复制完整历史对话；
4. Maven 和测试命令使用安静模式，只保留失败摘要和最终统计；
5. 开始任务前明确 Token 预算，并启用可查询的用量跟踪；
6. 对纯文档或机械重命名使用主智能体直接处理。

对于涉及依赖替换、配置优先级、权限上下文和测试有效性的任务，仍建议保留独立审查，因为本次多项重要问题正是由独立审查发现的。
