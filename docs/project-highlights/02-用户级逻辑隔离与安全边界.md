# 用户级逻辑隔离与安全边界

> 返回总览：[项目亮点与面试指南](../项目亮点与面试指南.md)

## 1. 能力定位

当前系统实现的是用户级逻辑隔离，不是完整的企业级 SaaS 多租户：

- 智能助手项目、会话、消息、动作和 Checkpoint 按 username 隔离。
- Mem0 记忆按 username 检索和写入。
- 检测任务对普通用户按 created_by 过滤，管理员可跨用户访问。
- 员工、设备、模型等企业资源仍是共享数据。
- 数据库尚未统一引入 tenant_id。

### 1.1 术语与字段翻译

| 英文术语或字段 | 中文名称 | 含义 |
| --- | --- | --- |
| SaaS | 软件即服务 | 多个组织通过网络共同使用一套软件平台 |
| tenant / tenant_id | 租户 / 租户编号 | 企业级多租户场景中的组织隔离标识；当前项目尚未统一引入 |
| JWT | JSON Web 令牌 | 用户登录后携带的签名身份凭证 |
| JwtAuthenticationFilter | JWT 认证过滤器 | 校验令牌并提取用户身份和权限 |
| SecurityContext | 安全上下文 | 保存本次请求中已经认证的可信用户身份 |
| Controller | 接口控制层 | 接收请求，但不信任客户端自行声明的用户编号 |
| Service | 业务服务层 | 将当前用户身份加入数据查询和修改条件 |
| username | 用户名 | 当前智能助手项目、会话和记忆的逻辑隔离键 |
| created_by | 创建人 | 检测任务所属用户字段 |
| userId | 用户标识 | Mem0 长期记忆使用的用户隔离键 |

## 4. 核心字段

| 表 | 字段 | 安全作用 |
| --- | --- | --- |
| users | id | 不可变用户主键，但当前业务授权尚未统一使用 |
| users | username | 当前登录标识和主要隔离键 |
| users | role | ADMIN 或 OPERATOR |
| chat_project | username | 项目归属 |
| chat_session | username | 会话、消息、动作和 Checkpoint 的归属根 |
| detection_task | created_by | 检测任务创建者 |
| detection_task | session_id | Agent 创建任务时关联会话 |

## 5. 核心代码

### 5.1 Controller 不接受客户端用户身份

~~~java
public SseEmitter streamMessage(
        Authentication authentication,
        ChatSendMessageRequest request) {
    return agentOrchestratorService.streamUserMessage(
            authentication.getName(), request);
}
~~~

username 从 Authentication 获取，而不是从 request 获取。

### 5.2 会话组合条件校验

~~~java
ChatSession session = chatSessionMapper.selectOne(
        new LambdaQueryWrapper<ChatSession>()
                .eq(ChatSession::getSessionId, sessionId)
                .eq(ChatSession::getUsername, username)
                .last("limit 1"));
if (session == null) {
    throw new BusinessException("会话不存在");
}
~~~

session_id 与 username 在同一 SQL 条件中出现，避免先返回其他用户记录再做判断。

### 5.3 检测任务访问策略

~~~java
if (accessPolicy.isAdmin(authentication)) {
    return;
}
if (authentication == null
        || !StringUtils.hasText(authentication.getName())) {
    wrapper.apply("1 = 0");
    return;
}
wrapper.eq(DetectionTask::getCreatedBy,
        authentication.getName());
~~~

普通用户增加 created_by 条件；管理员不增加；身份缺失时查询结果强制为空。

## 6. 会话子资源保护

消息和待确认动作本身没有 username 字段，安全边界来自父会话：

~~~mermaid
flowchart LR
    REQUEST[访问消息或动作] --> OWNER[校验 sessionId + username]
    OWNER -->|通过| CHILD[查询 chat_message / chat_pending_action]
    OWNER -->|失败| NOTFOUND[返回不存在]
~~~

删除、重命名、置顶、归档、查看消息、读取 Checkpoint 和确认动作前都应先校验会话归属。

## 7. 风险与演进

| 当前设计 | 风险 | 改进 |
| --- | --- | --- |
| username 作为隔离键 | 改名影响关联 | JWT 携带 users.id |
| 没有 tenant_id | 无法隔离多个企业 | organization + tenant_id |
| 各 Service 手工加条件 | 可能遗漏过滤 | TenantContext + 数据访问拦截 |
| 资源数据共享 | 无部门数据范围 | RBAC + DataScope |
| 本地限流按 username | 双实例状态不共享 | Redis 分布式限流 |

## 8. 正确的面试表述

可以说：“系统实现了智能助手和检测任务的用户级逻辑隔离，身份从 JWT 进入服务端上下文，查询使用资源编号与用户名组合条件。”

不能说：“系统已经实现完整企业级 tenant_id 多租户。”当前没有组织、租户成员关系或跨租户切换模型。

## 9. 面试问答

### 为什么跨用户访问返回不存在，而不是无权限？

返回不存在可以减少资源存在性泄露，攻击者无法通过响应差异枚举其他用户的 sessionId。

### username 作为隔离键有什么问题？

username 可变且带业务语义。更稳妥的方案是使用不可变 users.id 作为用户租户键，并把 username 仅用于展示和审计。
