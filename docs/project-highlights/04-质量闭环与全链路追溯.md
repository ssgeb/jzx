# 质量闭环与全链路追溯

> 返回总览：[项目亮点与面试指南](../项目亮点与面试指南.md)

## 1. 章节目标

本章讲解检测结果如何进入质检分派、人工复核、缺陷处置、返工和复检流程，以及系统如何利用数据库字段还原某个任务、批次或工单的完整质量历史。

### 1.1 字段与状态翻译

| 英文术语或状态 | 中文名称 | 含义 |
| --- | --- | --- |
| taskId / task_id | 任务编号 | 标识一次检测业务任务 |
| workflow_uuid | 工作流编号 | 串联智能助手、检测和质量追溯流程 |
| status / stage | 技术状态 / 技术阶段 | 描述上传、推理和结果处理的执行进度 |
| flow_status | 业务流转状态 | 描述复核、处置、返工和复检进度 |
| REVIEWING | 复核中 | 已分派质检员，等待人工确认 |
| CONFIRMED | 已确认 | 人工已经确认检测结论 |
| RELEASED | 已放行 | 产品允许进入后续生产或交付环节 |
| REWORK_REQUIRED | 需要返工 | 产品需要修复后重新进入质检流程 |
| RECHECK_REQUIRED | 需要复检 | 产品需要再次执行检查 |
| HOLD | 隔离 | 暂停流转并隔离问题产品 |
| SCRAPPED | 已报废 | 产品被判定无法继续使用 |
| PENDING_REVIEW | 等待复核 | 返工完成，等待再次人工复核 |

## 3. 状态机

~~~mermaid
stateDiagram-v2
    [*] --> UPLOADING
    UPLOADING --> PENDING_DETECTION: 上传完成
    PENDING_DETECTION --> COMPLETED: 全部检测成功
    PENDING_DETECTION --> PARTIAL_FAILED: 部分图片失败
    PENDING_DETECTION --> FAILED: 任务失败
    COMPLETED --> REVIEWING: 分派质检
    PARTIAL_FAILED --> REVIEWING: 分派质检
    REVIEWING --> CONFIRMED: 人工复核
    CONFIRMED --> RELEASED: RELEASE
    CONFIRMED --> REWORK_REQUIRED: REWORK
    CONFIRMED --> RECHECK_REQUIRED: RECHECK
    CONFIRMED --> HOLD: HOLD
    CONFIRMED --> SCRAPPED: SCRAP
    REWORK_REQUIRED --> PENDING_REVIEW: 返工完成
    REWORK_REQUIRED --> RECHECK_REQUIRED: 返工后要求复检
    RECHECK_REQUIRED --> REVIEWING: 重新分派
~~~

status/stage 主要描述技术执行状态；flow_status 描述质量业务流转。文档和接口不能把两组状态混为一个字段。

## 4. 核心字段

| 阶段 | 字段 | 说明 |
| --- | --- | --- |
| 任务标识 | task_id、workflow_uuid | 对外任务号和工作流号 |
| 生产追溯 | batch_no、work_order_no | 批次和工单查询主线 |
| 模型快照 | model_id、model_version、threshold | 固化本次任务使用的模型 |
| 分派 | quality_station、assignee、assigned_at、due_at | 质检站点、责任人和截止时间 |
| 复核 | review_status、review_conclusion、reviewer、reviewed_at | 人工判断和责任人 |
| 误报校正 | confirmed_defect_count、false_positive_count | 模型结果的人工校正 |
| 处置 | disposition_status、disposition_action、disposed_at | 放行、返工、复检、隔离或报废 |
| 返工 | rework_result、rework_operator、rework_completed_at | 返工结果与执行人 |
| 缺陷证据 | defect_evidence_json、primary_defect_type | 缺陷框、置信度、位置和类型 |

## 5. 缺陷证据结构

~~~json
{
  "imageKey": "source/example.jpg",
  "previewKey": "result/example.jpg",
  "defectType": "CRACK",
  "confidence": 0.972,
  "severity": "CRITICAL",
  "position": "handle-neck",
  "bbox": {"x": 132, "y": 100, "w": 108, "h": 70}
}
~~~

结构化证据让质量人员能够从缺陷统计回到具体图片和标注位置。

## 6. 核心代码

### 6.1 质检分派

~~~java
task.setQualityStation(request.getQualityStation().trim());
task.setAssignee(request.getAssignee().trim());
task.setAssignmentRemark(request.getAssignmentRemark());
task.setAssignedAt(now);
task.setDueAt(parseDateTime(request.getDueAt(), null));
task.setFlowStatus("REVIEWING");
~~~

只有 COMPLETED 或 PARTIAL_FAILED 的检测任务可以进入质检分派。

### 6.2 人工复核

~~~java
task.setReviewStatus("REVIEWED");
task.setReviewConclusion(reviewConclusion);
task.setSeverityLevel(severityLevel);
task.setConfirmedDefectCount(confirmedDefectCount);
task.setFalsePositiveCount(falsePositiveCount);
task.setReviewer(resolveCurrentUsername());
task.setReviewedAt(LocalDateTime.now());
task.setFlowStatus("CONFIRMED");
~~~

复核不会覆盖原始模型证据，而是额外保存人工结论、误报数和确认缺陷数。

### 6.3 缺陷处置

~~~java
task.setDispositionStatus("DISPOSED");
task.setDispositionAction(action);
task.setDispositionOperator(resolveCurrentUsername());
task.setDisposedAt(now);
task.setRecheckRequired(
        "RECHECK".equals(action)
        || Boolean.TRUE.equals(request.getRecheckRequired()));
task.setFlowStatus(resolveDispositionFlowStatus(
        action, task.getRecheckRequired()));
~~~

真实动作是 RELEASE、REWORK、RECHECK、HOLD、SCRAP，分别进入 RELEASED、REWORK_REQUIRED、RECHECK_REQUIRED、HOLD、SCRAPPED。

### 6.4 返工回填

~~~java
task.setReworkResult(request.getReworkResult().trim());
task.setReworkOperator(resolveCurrentUsername());
task.setReworkCompletedAt(now);
task.setReviewStatus("PENDING");
task.setFlowStatus(recheckRequired
        ? "RECHECK_REQUIRED" : "PENDING_REVIEW");
~~~

返工后重新进入复核或复检，而不是直接把任务标记为质量闭环完成。

## 7. 追溯时间线

~~~mermaid
flowchart LR
    C[CREATED] --> D[DISPATCHED]
    D --> F[FINISHED]
    F --> A[ASSIGNED]
    A --> R[REVIEWED]
    R --> P[DISPOSED]
    P --> W[REWORK_COMPLETED]
~~~

查询入口支持 taskId、workflowUuid、batchNo 和 workOrderNo。

## 8. 可靠性与约束

- 最终状态任务不能被无条件重新分派。
- 已处置任务不能重复复核。
- RELEASE 仅允许适合放行的复核结论。
- 任务、图片、模型和人员信息使用快照字段保证历史可解释。
- 质量队列使用 flow_status、review_status、disposition_status 组合筛选。

### 为什么模型检测后还要人工复核？

工业质检存在误报和漏报风险。模型提高筛查效率，人工复核负责最终质量判定，误报数还可反向用于模型评估。

### 如何回答“质量问题怎么追溯”？

以任务、批次和工单为主线，固化模型版本、检测证据、复核结论、处置动作、返工结果、操作人和时间，从而重建完整时间线。
