# 多智能体聊天助手总体设计方案

## 1. 文档目标

本文档用于定义当前项目引入多智能体聊天助手后的总体设计方案，目标是让用户尽量减少传统页面操作，只需要通过统一聊天入口与系统对话，即可完成检测、查询、报表和基础运维排障等任务。

本文档聚焦以下问题：

- 聊天助手以什么交互形态接入系统
- 多智能体体系应该如何拆分，且总数不超过 5 个 Agent
- 主 Agent 与专业 Agent 的职责边界如何划分
- 主 Agent 如何根据用户意图进行路由和编排
- 哪些任务可以直接执行，哪些任务必须确认
- 第一阶段应该优先落地哪些能力

本文档面向当前项目本身，默认多智能体能力第一版内嵌到当前 Java 后端中实现。

---

## 2. 设计结论

本项目的多智能体方案确定为：

1. 系统只提供一个全局聊天助手入口
2. 聊天界面在所有页面保持一致，共用同一套会话内容
3. 聊天助手不依赖页面上下文注入，主要依据用户输入和历史对话理解任务
4. 后端采用 `1 个主 Agent + 4 个专业 Agent` 的结构，总数不超过 5 个
5. 主 Agent 负责意图识别、任务拆解、确认和结果汇总
6. 专业 Agent 负责执行具体业务能力
7. 第一版采用“半自动执行”模式：查询类任务直接执行，关键动作执行前必须确认

这是当前项目最适合的总体方案，兼顾可落地性、可扩展性和用户体验。

---

## 3. 整体交互方案

### 3.1 统一入口

系统只保留一个聊天助手入口，建议放在：

- 全局右下角悬浮入口
- 点击后展开右侧抽屉式聊天面板

用户无论当前位于检测页、设备页、报表页还是其他页面，都通过这一入口与系统对话。

### 3.2 统一界面

所有页面共用同一个聊天界面和同一条会话链路，不为每个导航页面单独设计独立助手。

这样做的原因是：

- 用户心智更简单，不需要判断“应该在哪个页面找哪个助手”
- 任务天然可能跨模块，不适合按页面切割助手
- 前端只需要维护一套聊天 UI
- 后端只需要维护一套统一对话入口

### 3.3 不依赖页面上下文

第一版不自动读取当前页面状态、当前记录或当前筛选条件，不做页面上下文注入。

聊天助手主要依赖以下信息理解任务：

- 用户当前输入
- 历史聊天记录
- 需要时主动向用户追问的补充信息

这样设计的原因是：

- 架构更简单，前端改造量更小
- 避免因为页面状态误判任务目标
- 让聊天助手成为真正独立的系统入口

---

## 4. 多智能体总体架构

整体架构采用 `1 + 4` 模式：

1. `主 Agent`
2. `检测 Agent`
3. `资源 Agent`
4. `报表 Agent`
5. `运维 Agent`

其中：

- 用户只直接面对主 Agent
- 其他 4 个 Agent 属于后台执行层
- 用户不会直接感知多个 Agent 的内部切换

### 4.1 架构选择原因

相比“单助手 + 大量工具”的方案，这种拆分方式更适合当前项目，因为：

- 检测任务本身已经足够独立，值得单独建 Agent
- 设备、人员、模型管理可先统一归为资源管理，不需要拆得过细
- 报表和运维有清晰边界，适合独立为专业能力
- 结构清晰，后续扩展不会快速失控

相比“主 Agent + 工作流引擎 + 多 Agent”的重型方案，这种方式第一版更容易落地。

---

## 5. Agent 职责边界

### 5.1 主 Agent

主 Agent 是唯一对话入口，职责只包括：

- 理解用户自然语言
- 识别任务意图
- 判断任务是否信息充分
- 将任务路由到合适的专业 Agent
- 对复合任务进行拆解和顺序编排
- 在关键动作前向用户发起确认
- 汇总多个 Agent 的执行结果并组织成最终回复

主 Agent 不直接调用底层业务接口，不直接承担具体业务执行。

### 5.2 检测 Agent

检测 Agent 负责所有与检测链路相关的业务：

- 发起单图或批量检测任务
- 确认模型、阈值和输入范围
- 调用检测任务创建接口
- 跟踪检测任务状态与进度
- 获取检测结果摘要、预览图和结果文件
- 解释检测结果和失败原因
- 处理与 OSS 上传、远程 FastAPI 调用相关的检测业务流程

检测 Agent 不负责设备管理、人员管理和系统级排障。

### 5.3 资源 Agent

资源 Agent 合并处理设备、人员和模型三类资源管理能力，职责包括：

- 查询设备、人员、模型信息
- 新增、修改、删除资源类记录
- 处理资源之间的关联查询
- 回答资源可用性、状态、版本等问题
- 为检测 Agent 提供模型或资源参考信息

之所以不拆成“设备 Agent / 人员 Agent / 模型 Agent”，是因为第一版没必要把管理类能力拆得太细，合并后更容易维护。

### 5.4 报表 Agent

报表 Agent 负责统计、分析和汇总类任务，职责包括：

- 回答首页统计类问题
- 汇总检测结果趋势
- 按设备、人员、模型等维度进行统计分析
- 生成日报、周报和异常汇总
- 将结构化数据转成用户易读的结论

报表 Agent 主要负责“解释和汇总”，不直接发起检测任务，也不直接改业务数据。

### 5.5 运维 Agent

运维 Agent 负责健康检查和故障排查，职责包括：

- 分析 OSS 上传失败问题
- 分析远程 FastAPI 调用异常
- 分析检测任务卡住、超时或失败的原因
- 给出重试、补偿或人工介入建议
- 查询系统关键链路的健康状态

第一版中，运维 Agent 先聚焦在“检测链路排障”，不追求覆盖全部系统运维能力。

---

## 6. 主 Agent 路由机制

主 Agent 的路由依据不是“当前页面”，而是“用户意图”。

### 6.1 意图分类

主 Agent 需要先将输入分类为以下几种类型：

- 检测任务
- 资源管理
- 报表分析
- 运维排障
- 复合任务
- 信息不足任务

### 6.2 单一意图路由

如果用户意图属于单一任务，则直接路由到对应 Agent：

- 检测类请求 -> 检测 Agent
- 设备 / 人员 / 模型类请求 -> 资源 Agent
- 统计与总结类请求 -> 报表 Agent
- 异常与失败排查类请求 -> 运维 Agent

### 6.3 复合任务编排

如果用户请求包含多个步骤，主 Agent 负责拆解并按顺序调用多个 Agent。

例如：

`用最新模型检测今天上传的图片，再把异常结果做成简报`

处理流程为：

1. 主 Agent 识别出“检测 + 报表”复合任务
2. 先调用检测 Agent 完成任务创建与进度跟踪
3. 检测完成后再调用报表 Agent 汇总异常结果
4. 最后由主 Agent 汇总结果回复用户

### 6.4 信息不足时的处理

如果任务缺少关键参数，主 Agent 不能直接执行，而是先追问。

例如：

- “帮我开始检测”
- “给我生成报表”
- “帮我改一下设备信息”

这类请求如果缺少目标范围、模型、时间区间或具体对象，主 Agent 应先补问，再路由执行。

### 6.5 统一汇总输出

无论底层调用了多少个专业 Agent，最终都由主 Agent 统一组织成用户可读回复。

用户只看到一个聊天助手，不直接看到多个 Agent 的中间结果。

---

## 7. 执行与确认机制

第一版采用“半自动执行”模式。

### 7.1 可直接执行的任务

以下低风险、只读类任务可以直接执行：

- 查询检测任务状态
- 查询设备、人员、模型信息
- 查询统计结果
- 查看失败原因
- 查看历史记录

### 7.2 必须确认后执行的任务

以下会产生业务动作或资源消耗的任务，必须在执行前由主 Agent 向用户确认：

- 发起批量检测任务
- 创建、修改、删除资源记录
- 生成正式报表
- 重试失败任务
- 触发外部服务调用且可能产生明显资源消耗的动作

### 7.3 第一版不建议自动放开的任务

以下高风险动作第一版不建议完全自动化：

- 删除核心业务数据
- 批量覆盖系统配置
- 修改安全相关参数
- 跳过确认直接执行大范围写操作

对于这类操作，第一版更适合由助手提供建议或引导到人工操作。

---

## 8. 典型任务流程

### 8.1 检测类任务

示例：

`帮我开始一批门把手图片检测`

处理流程：

1. 主 Agent 识别为检测任务
2. 如果信息不足，追问模型或图片来源
3. 主 Agent 请求用户确认执行
4. 检测 Agent 调用检测任务接口
5. 检测 Agent 跟踪任务状态
6. 主 Agent 将进度和最终结果回复用户

### 8.2 资源类任务

示例：

`查一下设备 A 最近的使用记录`

处理流程：

1. 主 Agent 识别为资源查询
2. 路由到资源 Agent
3. 资源 Agent 调用资源查询接口
4. 主 Agent 输出结构化结果和摘要说明

### 8.3 报表类任务

示例：

`生成今天的检测摘要`

处理流程：

1. 主 Agent 识别为报表任务
2. 如时间范围不清晰则先补问
3. 路由到报表 Agent
4. 报表 Agent 汇总统计结果
5. 主 Agent 输出简报内容

### 8.4 运维类任务

示例：

`这个检测任务为什么失败了`

处理流程：

1. 主 Agent 识别为运维排障任务
2. 路由到运维 Agent
3. 运维 Agent 查询错误信息、外部调用记录或关键状态
4. 主 Agent 返回故障原因和建议动作

---

## 9. 第一阶段落地范围

第一阶段不追求把 5 个 Agent 的所有能力一次做满，而是优先做“对话式业务闭环”。

### 9.1 第一阶段重点建设

建议第一阶段重点实现：

- 主 Agent
- 检测 Agent
- 资源 Agent
- 基础版报表 Agent
- 基础版运维 Agent

### 9.2 第一阶段用户可完成的任务

#### 检测类

- 帮我开始一批检测
- 用最新模型检测
- 看看进度
- 把结果总结一下

#### 资源类

- 查设备信息
- 查人员信息
- 看当前可用模型
- 修改某条资源记录

#### 报表类

- 生成今天的检测摘要
- 看最近 7 天异常分布
- 汇总某模型下的检测结果

#### 基础运维类

- 这个任务为什么失败了
- OSS 上传有没有问题
- 远程检测服务是否异常

### 9.3 第一阶段暂不深做的内容

第一阶段暂不建议投入过深的能力包括：

- 复杂多轮工作流自动回滚
- 非检测链路的深度系统运维
- 大规模跨系统协同自动化
- 高风险写操作完全自动执行

这些能力可以在第一版稳定后再逐步补强。

---

## 10. 后端落地建议

由于当前项目的业务后端已经承担了检测任务编排、OSS 对接和远程 FastAPI 调用，第一版多智能体能力建议内嵌到当前 Java 后端中。

建议后端分为三层：

1. `Chat API 层`
   - 提供统一聊天接口
   - 管理会话、消息、确认状态

2. `Agent 编排层`
   - 实现主 Agent 的路由与调度
   - 管理专业 Agent 的输入输出协议

3. `工具 / 服务调用层`
   - 调用检测、资源、报表、运维相关现有服务接口

这样做的好处是：

- 不破坏现有业务接口体系
- 可以逐步将现有服务能力包装为 Agent 工具
- 后续如果要把 Agent 能力独立成单独服务，也有清晰分层可迁移

---

## 11. 前端落地建议

前端第一版建议仅增加一套统一聊天 UI，不为每个页面做独立助手实现。

建议包括：

- 全局悬浮聊天按钮
- 右侧抽屉式聊天面板
- 统一消息列表
- 用户确认卡片
- 任务执行状态消息
- 结果摘要卡片

前端重点不在复杂智能，而在于把以下交互做清楚：

- 用户发问
- 助手追问
- 用户确认
- 任务执行中
- 结果返回

---

## 12. 结论

本项目最合适的多智能体设计方案是：

- 一个统一的全局聊天助手入口
- 统一聊天界面和共享会话
- 不依赖页面上下文注入
- 后端采用 `1 个主 Agent + 4 个专业 Agent`
- 主 Agent 负责理解、拆解、确认、汇总
- 专业 Agent 负责检测、资源、报表、运维执行
- 第一版采用半自动执行策略
- 第一阶段优先做“对话式检测与查询闭环”

这套方案既能明显减轻用户操作，又不会在第一版把系统做得过重，是当前项目最平衡、最适合落地的方向。
