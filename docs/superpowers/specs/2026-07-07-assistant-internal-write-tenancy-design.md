# 智能助手内部写链路租户加固设计

## 背景

智能助手已使用 `users.id` 隔离项目、会话和 Mem0 记忆，但部分内部服务仍只凭 `sessionId` 写入消息、checkpoint 和待确认动作。同步控制器已校验登录用户，然而异步检测任务、StateGraph 节点和后续新增调用者可能绕过入口校验。需要把租户约束下沉到内部写接口，避免跨用户串写。

## 目标

- 所有助手会话写操作同时校验 `user_id` 和 `session_id`。
- 异步任务显式携带不可变 `TenantContext`，不依赖线程本地安全上下文。
- checkpoint、消息和待确认动作继承所属会话的租户边界。
- 删除 Mem0 仅按字符串用户标识调用的旧接口，防止新代码退回弱作用域。
- 保持企业业务数据共享，不扩大租户隔离范围。

## 非目标

- 不修改检测任务、设备、员工、模型等共享业务数据模型。
- 不改变前端接口格式。
- 不为消息和待确认动作重复增加 `user_id` 字段；它们通过会话外键继承归属。
- 不引入新的消息队列协议或分布式追踪系统。

## 方案

### 1. 租户感知的内部接口

`ChatSessionService` 的消息、checkpoint 和待确认动作接口增加 `TenantContext` 参数。实现层先用 `(session_id, user_id)` 查询所属会话；不存在时统一抛出 404，再执行读写。公开的会话管理接口继续从认证主体取得用户身份。

租户校验集中到一个内部方法，避免各写方法自行拼接不同条件。用户名只用于展示和兼容日志，不作为授权依据。

### 2. 同步编排链路

`AgentOrchestratorServiceImpl` 在请求进入时从 `TenantPrincipal` 生成 `TenantContext`，并在整个编排过程中传递。追加用户消息、助手消息、读写 checkpoint、创建或更新待确认动作时均使用该上下文。

SSE 在提交工作线程前捕获 `TenantContext`。安全上下文传播仍保留给其他依赖方，但租户授权不依赖工作线程中的 `ThreadLocal`。

### 3. StateGraph 与 checkpoint

StateGraph 状态加入仅供服务端使用的租户用户 ID。`MySqlCheckpointer` 的保存和加载使用租户感知接口；恢复和确认动作必须同时匹配用户与会话。

图节点需要写入待确认动作时，从 `AgentState` 恢复 `TenantContext`。缺少或非法租户 ID 时立即失败，不降级为仅按会话号操作。

### 4. 异步检测回写

启动异步检测时，将 `TenantContext` 与 `sessionId` 一起封装进任务参数。后台完成、失败和进度回写均使用显式租户上下文。异步任务不得通过当前线程的认证信息推断租户。

检测任务创建请求携带 `sessionId` 时，必须在持久化关联前校验该会话属于当前登录用户。对于稍后由检测引擎触发、已脱离原请求生命周期的最终完成通知，系统通过已校验并持久化的会话关联读取会话所有者，构造 `TenantContext` 后再调用租户感知写接口。这个恢复入口仅供系统回调使用，不接受前端直接调用。

对于不来自智能助手会话的检测任务，`sessionId` 为空时保持现有行为，不执行会话回写。

### 5. Mem0 接口收敛

删除 `addMemory(String userId, ...)`、`searchMemories(String userId, ...)` 和对应异步重载，只保留要求 `TenantContext + sessionId` 的接口。Mem0 继续使用：

- `user_id = doorhandlecatch:user:{users.id}`
- `app_id = doorhandlecatch`
- `run_id = sessionId`

## 错误处理

- 会话不存在或不属于当前租户：404，避免泄露资源存在性。
- 缺少租户上下文：401。
- 认证用户名与显式上下文不一致：403。
- 异步回写发现会话已删除或不属于租户：记录结构化警告并停止回写，不重试到其他会话。
- Mem0 不可用：保持当前降级策略，不影响主对话结果。

## 兼容与迁移

这是 Java 内部接口调整，不改变 HTTP API。一次性更新所有编译期调用点，利用编译器确保没有遗漏旧签名。数据库无需新增迁移。

历史 checkpoint 若缺少租户用户 ID，只允许在已通过 `(session_id, user_id)` 会话校验后加载，并在下一次保存时补齐租户字段。

## 测试

- 服务单元测试：同一 `sessionId` 在错误 `userId` 下无法写消息、checkpoint 或待确认动作。
- 编排测试：同步与 SSE 都传递相同 `TenantContext`。
- StateGraph 测试：确认动作和恢复 checkpoint 不接受缺失或错误租户。
- 异步测试：工作线程安全上下文为空时，仍使用任务携带的租户正确回写。
- Mem0 编译与单测：仅允许完整三维作用域。
- 回归测试：执行完整 Maven 测试和 Mem0 Python 租户测试。

## 验收标准

1. 代码中不存在仅凭 `sessionId` 写入助手消息、checkpoint 或待确认动作的生产调用。
2. 代码中不存在 Mem0 的字符串用户标识公开重载。
3. 跨租户写入测试稳定返回 404，且目标数据不变。
4. 异步上传任务不依赖 `SecurityContextHolder` 获取租户；延迟系统回调从已校验的持久化会话关联恢复租户。
5. 全量 Java 测试与新增 Python 租户测试通过。
