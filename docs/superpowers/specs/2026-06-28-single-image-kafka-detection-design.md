# 单图 Kafka 检测链路设计

## 目标

将单图检测统一接入现有 OSS 与 Kafka 远程推理链路，停止调用本地同步 ONNX 接口，杜绝模型输出解析失败后返回固定 `20%` 的假结果。

## 数据流

1. 用户在单图检测模块选择图片、模型和置信度阈值。
2. 前端调用 `POST /api/detection/tasks` 创建 `taskType=SINGLE` 的任务，提交单个文件元数据、模型 ID、阈值和单图采集信息。
3. 后端返回任务编号、OSS 对象键和预签名上传地址。
4. 前端通过现有 `/oss-upload` 代理将原图直传 OSS。
5. 前端调用 `POST /api/detection/tasks/{taskId}/uploaded` 确认上传完成。
6. 后端异步发布 `detection.task.created` Kafka 事件，远程 Python Worker 下载原图、执行 YOLO 推理并上传标注结果。
7. 前端轮询任务进度和结果接口，将第一张标注图、缺陷类别、置信度和任务编号显示在单图结果框。

## 前端边界

- 单图检测不再调用 `/api/image-detection/upload`。
- 新增聚焦单图场景的 API/组合函数，负责创建任务、OSS 上传、确认上传和轮询，不复制页面级业务逻辑。
- 结果框展示 `准备上传`、`上传中`、`Kafka 排队`、`远程推理`、`完成`、`失败` 六类状态。
- 任务完成后保留所选原图、任务编号和远程结果；重新选择图片时清理上一轮显示状态。
- 不执行本地 ONNX 回退，不把模拟概率当作远程结果。

## 后端边界

- 复用现有任务创建、上传确认、Kafka 发布和结果回写接口，不新增同步等待 Kafka 的控制器。
- `SINGLE` 任务必须保持任务类型，并纳入检测记录、追溯和模型使用统计。
- Kafka 发布失败、Worker 回写失败和任务超时必须进入失败状态并保存可读错误信息。
- 本次不删除旧 `/api/image-detection/upload` 接口，避免破坏其他调用方；活动检测工作台不再使用它。

## 失败处理

- OSS 上传失败：停止调度，结果框显示上传失败并允许重新检测。
- Kafka Broker 不可达：任务标记失败，显示 Kafka 调度失败和任务编号。
- Worker 超时：前端停止轮询，显示远程检测超时；数据库任务保留原状态供运维排查。
- 远程结果缺少标注图或置信度：显示“远程结果不完整”，不生成默认 `20%`。

## 当前运行约束

当前配置的 Kafka Broker 为 `10.20.144.118:9092`，运行日志显示连接失败。代码改造完成后，只有 Broker 与远程 Worker 可访问时才能得到成功检测结果；不可用时系统应明确失败，而不是退回本地假结果。

## 验证

- 前端契约测试确认活动页面不再引用 `detectSingleImage`。
- 单元测试覆盖单图任务创建、OSS 上传确认、轮询成功、Kafka 失败、超时和结果不完整。
- 使用可用 Kafka/Worker 时完成一次端到端单图检测，验证任务类型、结果图、类别、置信度和任务追溯。
- Kafka 不可用时验证界面显示明确错误且不存在固定 `20%` 结果。
