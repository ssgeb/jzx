# 单企业全员共享数据设计

## 1. 背景与目标

DoorHandleCatch 仅服务一个企业。系统包含多个登录用户，但不存在企业租户之间的数据隔离需求。所有已登录用户应访问同一份企业业务数据，并拥有完全相同的系统操作权限。

本设计目标：

- 保留登录认证，未登录用户不能访问业务接口。
- 所有已登录用户拥有相同的查询、新增、修改、删除、审核和管理能力。
- 所有业务数据及聊天数据在用户之间共享。
- 保留创建人和操作者信息用于审计，不再用于访问控制。
- 不引入 `tenant_id`，不复制现有数据，不改变现有业务主键。

## 2. 范围

共享范围包括：

- 检测任务、检测记录、原图、标注图、检测结果和 OSS 资源。
- 设备、设备告警、设备使用记录和员工信息。
- 模型、模型评估、发布、回滚和操作记录。
- 质检队列、复核、处置、返工、批次与工单追溯。
- 仪表盘、统计和报表。
- 聊天项目、聊天会话、消息、待确认操作和 Agent checkpoint。

本次不建设多企业租户、部门数据权限、记录级所有权隔离或差异化角色权限。

## 3. 认证与授权模型

系统采用“认证后进入统一企业数据空间”的模型：

```text
登录认证 -> 建立当前用户身份 -> 访问统一企业数据
                              -> 写操作记录审计日志
```

- Spring Security 继续要求业务接口必须经过 JWT 认证。
- `ADMIN`、`OPERATOR`、`REVIEWER` 等角色不影响业务权限。
- `users.role` 暂时保留，避免破坏现有接口、前端和存量数据兼容性。
- `created_by`、`username` 等归属字段继续记录最初创建者，但不得作为列表、详情、修改或删除的授权条件。
- 未登录请求返回 HTTP `401`；已登录用户不会因角色或创建人不同收到 `403`。

## 4. 数据模型

现有业务表继续保存单份共享数据，不新增 `tenant_id`。

- `detection_task.created_by` 表示任务创建者。
- `chat_session.username` 和 `chat_project.username` 表示最初创建者。
- 员工编号、设备编号、模型名称与版本继续保持企业内唯一。
- 现有外键、主键和业务唯一约束保持不变。

新增统一操作审计表：

```sql
CREATE TABLE operation_audit_log (
    id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    operator        VARCHAR(64) NOT NULL,
    action          VARCHAR(32) NOT NULL,
    resource_type   VARCHAR(64) NOT NULL,
    resource_id     VARCHAR(128),
    request_method  VARCHAR(16),
    request_path    VARCHAR(512),
    change_summary  JSON,
    result          VARCHAR(16) NOT NULL,
    client_ip       VARCHAR(45),
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_audit_operator_time (operator, created_at),
    INDEX idx_audit_resource_time (resource_type, resource_id, created_at),
    INDEX idx_audit_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

审计日志只保存变更摘要，不保存密码、JWT、Cookie、访问密钥、完整图片、模型文件或大段请求正文。审计日志不提供普通业务修改和删除接口。

## 5. 后端设计

### 5.1 数据访问

- 检测任务及检测记录查询移除 `created_by = 当前用户` 条件。
- `DetectionTaskAccessPolicy` 只确认当前请求已经认证，不比较任务创建人或角色。
- OSS 预览授权仍校验资源确实被系统任务引用，但不限制引用任务的创建者。
- 本地文件下载只要求登录，并继续使用安全路径解析防止目录穿越。
- 聊天项目、会话和消息查询移除当前用户名过滤；更新、移动、归档和删除也不检查创建者。
- 设备、员工、模型、统计和质检接口对所有登录用户保持一致。
- 所有大列表继续使用分页，避免共享数据增长后出现无界查询。

### 5.2 写操作与审计

新增 `OperationAuditService`，统一记录以下操作：

- 新增、修改和删除业务实体。
- 创建任务、续传、重新检测和任务状态流转。
- 复核、处置、返工和告警处理。
- 模型验证、发布、设为默认、禁用、归档和回滚。
- 聊天项目或会话的修改、归档和删除。

审计记录包含当前认证用户名、动作、资源类型、资源标识、请求路径、结果和经过脱敏的变更摘要。

业务更新继续使用事务。需要强一致审计的操作在同一事务中写入成功日志；失败日志在主事务回滚后通过独立事务写入。审计写入异常必须记录监控告警，不得暴露敏感请求内容。

### 5.3 并发与错误处理

共享写权限会提高并发冲突概率：

- 状态流转使用“主键 + 预期状态”条件更新。
- 可覆盖编辑的实体增加版本号或等效乐观锁条件。
- 受影响行数为零时返回 HTTP `409`，提示数据已被其他用户修改。
- 删除不存在的资源返回 `404`。
- 参数校验失败返回 `400`，认证失败返回 `401`。
- 高风险写操作保持幂等或通过状态条件防止重复执行。

## 6. 前端设计

- 登录和会话续期逻辑保持不变。
- 移除基于角色隐藏菜单、按钮或操作入口的逻辑。
- 所有用户看到相同的业务页面和操作能力。
- 列表和详情展示创建人、最后操作人及更新时间（数据存在时）。
- 删除模型、删除设备、模型回滚等高风险操作保留二次确认。
- 收到 `409` 时提示记录已变化，并引导刷新后重试，不静默覆盖。
- 不依赖前端权限隐藏实现安全控制；后端仍统一要求认证。

## 7. 迁移与发布

迁移不复制、不删除现有业务数据：

1. 新增 `operation_audit_log` 表。
2. 部署兼容现有字段的后端版本。
3. 移除角色和创建人访问过滤，接入审计与并发保护。
4. 部署移除角色界面限制的前端版本。
5. 使用两个普通账号执行共享数据验收。
6. 观察 `401`、`403`、`409`、审计写入失败和慢查询指标。

回滚时恢复原有角色和用户名过滤逻辑。新增审计表可安全保留，不需要在回滚中删除。

## 8. 验收测试

- 用户 A 创建的检测任务，用户 B 可以查询、修改、审核和删除。
- 用户 B 可以访问用户 A 创建的聊天项目、会话、消息和待确认操作。
- 任意已登录用户可以管理模型、设备、员工、告警和质检流程。
- 任意已登录用户可以预览任务引用的 OSS 文件和本地文件。
- 未登录请求仍返回 `401`。
- 用户角色与资源创建人不同不会产生 `403`。
- 每次关键写操作都生成包含正确操作者和资源标识的审计记录。
- 密码、JWT、Cookie、密钥和二进制内容不会进入审计日志。
- 两个用户同时更新同一记录时，后提交者收到 `409`，先提交的数据不会被静默覆盖。
- 迁移前的存量数据无需处理即可被所有登录用户访问。

## 9. 非目标

- 不支持多个企业或租户切换。
- 不支持用户级、部门级或角色级数据隔离。
- 不建设细粒度 RBAC/ABAC 权限配置中心。
- 不允许匿名访问业务数据。
- 不删除现有角色和创建人字段。
