## 一、项目定位与总体架构

本项目面向门把手工业视觉质检场景，将图片采集、模型推理、人工复核、缺陷处置、返工复检、质量追溯和智能助手整合为一套业务平台。系统不是单纯的模型演示，而是围绕“检测任务”建立完整的数据与状态闭环。

## 二、亮点一：Harness Agent 多智能体编排与分层记忆

### 2.1 解决的问题

传统大模型助手通常只有“拼接提示词—调用模型—返回文本”三步，难以安全处理真实业务。本项目将一次对话拆成状态图执行过程，使路由、参数收集、业务调用、人工确认和异常兜底都成为可观察节点。

### 2.2 Agent 模块图

~~~mermaid
flowchart LR
    INPUT[用户消息] --> CONTEXT[恢复 Checkpoint]
    CONTEXT --> RAG[检索 ChromaDB]
    RAG --> MEMORY[检索 Mem0]
    MEMORY --> ROUTER[RouterNode]

    ROUTER -->|参数不足| SLOT[SlotFillingNode]
    SLOT --> ROUTER

    ROUTER -->|检测业务| DET[DetectionAgentNode]
    ROUTER -->|资源业务| RES[ResourceAgentNode]
    ROUTER -->|报表业务| REP[ReportAgentNode]
    ROUTER -->|运维诊断| OPS[OpsAgentNode]
    ROUTER -->|修改操作| CONFIRM[HumanConfirmNode]

    CONFIRM -->|等待确认| SAVE[保存 Checkpoint 并中断]
    SAVE -->|confirm + resume| ROUTER

    DET --> RESPONSE[ResponderNode]
    RES --> RESPONSE
    REP --> RESPONSE
    OPS --> RESPONSE
    RESPONSE --> OUTPUT[SSE 文本或业务卡片]

    ROUTER -->|异常/超时/循环| FALLBACK[FallbackNode]
    FALLBACK --> OUTPUT
~~~

### 2.2.1 Agent 之间如何传递消息和沟通

本项目采用“中心路由 + 共享状态”的黑板模式。专业 Agent 不通过 Kafka、RPC 或 HTTP 互相发送消息，也不会自由对话；`RouterNode` 负责选择执行节点，各节点通过同一个 `AgentState` 读取和写入信息，`StateGraph` 再根据意图、阶段和确认状态决定下一步。

~~~text
用户消息
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  编排服务创建 AgentState                                     │
│  用户输入 + 身份 + 会话 + 历史消息 + RAG + Mem0              │
└──────────────────────────┬───────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│  RouterNode 写入路由决定                                     │
│  targetAgent + intent + slots + confirmationRequired         │
└──────────────────────────┬───────────────────────────────────┘
                           │
                 ┌─────────┼─────────┐
                 │         │         │
              参数不足    查询      修改
                 │         │         │
                 ▼         ▼         ▼
             槽位补全   专业 Agent  人工确认节点
                 │         │         │
                 └─────────┴────┬────┘
                                ▼
┌──────────────────────────────────────────────────────────────┐
│  将执行结果写回 AgentState                                   │
│  resultContent + resultType + intent + exitReason            │
└──────────────────────────┬───────────────────────────────────┘
                           │
                           ▼
             保存消息和 Checkpoint，通过 HTTP/SSE 返回
~~~

| 传递信息 | 作用 |
| --- | --- |
| `routeDecision`、`intent` | 告诉状态图应该选择哪个专业 Agent |
| `slots`、`missingSlots` | 在多轮对话中传递已收集和缺少的参数 |
| `recentMessages`、`summary` | 为下一轮路由和回答恢复上下文 |
| `ragContext` | 给专业 Agent 注入项目公共知识 |
| `confirmed`、`pendingActionId` | 支持修改操作的暂停和人工确认 |
| `resultContent`、`resultType` | 将专业 Agent 的执行结果交给编排服务和前端 |

MySQL Checkpoint 负责跨请求保存机器执行状态，聊天记录负责保存用户可见文本，Mem0 负责用户长期记忆。Kafka 只用于图像检测任务与 Python Worker 的异步通信，不参与 Agent 内部协作。当前各节点由 `CompiledGraph` 顺序执行，一次通常只选择一个专业 Agent，并非多 Agent 并行讨论或投票。

### 2.2.2 主 Agent 与子 Agent 如何协同

项目中的主 Agent 是逻辑编排层，不是另一个与子 Agent 自由聊天的大模型。`AgentOrchestratorServiceImpl` 负责请求入口和上下文，`CompiledGraph` 负责节点调度与检查点，`RouterNode` 负责决定目标；检测、资源、报表和运维节点是专业子 Agent。

~~~text
用户消息
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  逻辑主 Agent                                               │
│  Orchestrator：身份、会话、记忆和结果                        │
│  CompiledGraph：节点、条件边、守卫和 Checkpoint              │
│  RouterNode：intent、targetAgent 和 slots                    │
└──────────────────────────┬───────────────────────────────────┘
                           │ 同一个 AgentState
            ┌──────────────┼──────────────┬──────────────┐
            ▼              ▼              ▼              ▼
       检测子 Agent   资源子 Agent   报表子 Agent   运维子 Agent
            │              │              │              │
            └──────────────┴───────┬──────┴──────────────┘
                                   ▼
              写回 resultContent、resultType、exitReason
                                   │
                                   ▼
                    主 Agent 保存状态并通过 SSE 返回
~~~

子 Agent 之间没有点对点消息通道，也不会直接互相调用。需要协作的数据必须写入 `AgentState`，再由状态图的条件边交给下一个节点。这样可以统一记录路由、保存状态、限制权限，并在失败时定位到具体节点。

| 节点 | 职责 | 关键输入 | 关键输出 |
| --- | --- | --- | --- |
| RouterNode | 识别意图和目标智能体 | 用户消息、历史消息、RAG | intent、targetAgent |
| SlotFillingNode | 补齐任务编号、设备编号等槽位 | collectedSlots、missingSlots | 追问内容或可执行状态 |

### 2.6 核心代码：恢复短期上下文

文件：src/main/java/com/ruanzhu/doorhandlecatch/service/impl/AgentOrchestratorServiceImpl.java

~~~java
private AgentState buildStateWithContext(
        String sessionId,
        ChatSendMessageRequest request,
        String username) {
    AgentState state = AgentState.create(
            sessionId, request.getContent(), username);

    AgentState previous = checkpointer.load(sessionId);
    if (previous != null) {
        for (String key : CONTEXT_KEYS) {
            Object value = previous.get(key, Object.class);
            if (value != null) {
                state.set(key, value);
            }
        }
    }
    return state;
}
~~~

这里没有直接复用旧状态对象，而是创建本轮新状态，再选择性恢复上下文字段。这样可以保留槽位、会话阶段和最近消息，同时避免把上一轮临时结果无条件带入下一轮。

### 2.9 可靠性设计

Agent 安全不是只依赖提示词，而是由身份、状态图、数据库状态机和失败降级共同保证。

~~~text
身份/会话校验
      │
      ▼
参数是否完整？──否──> 只追问，不调用业务子 Agent
      │是
      ▼
查询还是修改？
  ┌───┴────┐
 查询     修改
  │        │
  │    保存 PENDING 动作并暂停
  │        │
  │    用户确认且条件抢占成功
  └───┬────┘
      ▼
专业子 Agent 执行
      │
      ▼
节点失败？──是──> 配置内重试 → Fallback → 保存失败状态
      │否
      ▼
保存 Checkpoint 并返回
~~~

| 安全与恢复措施 | 当前实现 |
| --- | --- |
| 租户隔离 | `TenantContext` 写入 `AgentState`，Checkpoint 按 `userId + sessionId` 读写 |
| 高风险确认 | `PENDING → EXECUTING → COMPLETED/FAILED`，条件更新防止重复确认 |
| 最大执行轮数 | 15 |
| 单轮最大执行时间 | 15000 ms |
| 单节点最大访问次数 | 4 |
| 同一路由最大重复次数 | 3 |
| Router 重试 | 配置声明 2 次并提供指数退避代码路径；尚缺独立重试映射回归测试，不作为强保证 |
| 异常兜底 | 写入错误和退出原因，进入 `FallbackNode`，不编造业务结果 |
| 中断恢复 | 每个节点后保存 MySQL Checkpoint，确认时从当前节点恢复 |

失败恢复分为三类：执行器为配置成可重试的节点提供指数退避代码路径；节点最终失败、节点不存在或运行守卫触发时进入 Fallback 并保存失败轨迹；等待确认的写操作保存 Checkpoint 后暂停，用户确认时重新校验租户，再注入 `confirmed=true` 从保存节点继续。当前尚缺按节点重试映射的独立回归测试，因此不能把 Router 必然重试两次作为已验收保证。

需要明确当前边界：普通对话读取 Checkpoint 失败会降级为新状态，高风险确认找不到 Checkpoint 时会直接失败关闭；Checkpoint 写入失败只记录警告，系统没有跨所有 Agent 的通用分布式事务与自动补偿机制，业务回滚仍依赖具体 Service 的事务边界。详细代码和恢复矩阵见[独立章节](project-highlights/01-Harness多智能体编排与分层记忆.md#34-agent-失败后如何恢复)。

### 2.10 面试表达

**问：为什么不用一个 Agent 直接处理所有业务？**

答：检测、资源、报表和运维依赖的数据源与参数不同。拆成专业智能体后，每个节点只依赖自己的服务，提示词更短，测试边界更清晰，修改某一业务不会扩大其他智能体的回归范围。

**问：聊天记录和 Checkpoint 有什么区别？**

答：聊天记录保存用户可见消息；Checkpoint 保存机器执行状态，包括当前节点、槽位、路由结果和待确认动作。只有聊天记录无法精确恢复被中断的工作流。

**问：主 Agent 和子 Agent 是通过什么方式通信的？**

答：不是通过 Kafka、RPC 或自由对话，而是通过共享的 `AgentState`。主 Agent 编排层写入身份、上下文和路由决定，状态图选择一个专业子 Agent；子 Agent 把结果写回状态，主 Agent 再统一持久化和输出。

**问：子 Agent 失败以后如何恢复？**

答：执行器设计了按节点配置的指数退避重试，但当前仍缺重试映射的独立回归测试；节点最终失败或守卫触发时会进入 Fallback，保存错误、退出原因和节点轨迹。等待人工确认的操作会先保存 MySQL Checkpoint，确认后按租户加载状态并从当前节点继续。项目不会在缺少 Checkpoint 时猜测原操作继续执行。

### 2.11 本地语音输入与中文识别

项目的语音功能是自动语音识别（ASR，即语音转文字），默认使用本地 `faster-whisper`，不是浏览器自带识别服务。浏览器只通过 `MediaRecorder` 录音，Spring Boot 完成文件校验和安全转发，Python FastAPI 服务负责加载 Whisper 模型并返回中文文本。

| 技术名词 | 中文说明 | 项目用途 |
| --- | --- | --- |
| ASR | 自动语音识别 | 把用户语音转换成文字 |
| MediaRecorder | 浏览器媒体录制接口 | 获取麦克风音频并生成上传文件 |
| faster-whisper | Whisper 高性能推理实现 | 在本机执行中文语音识别 |
| VAD | 语音活动检测 | 过滤静音片段 |
| INT8 | 8 位整数计算精度 | 降低 CPU 推理开销 |
| TTS | 语音合成 | 文字转语音；当前项目尚未实现 |

该设计的重点不是单纯增加一个麦克风按钮，而是把录音、识别和业务执行分开：识别服务可以独立替换，音频默认不出本机，识别错误也不会绕过人工确认直接进入智能体状态图。更完整的配置、核心代码和测试说明见[独立章节](project-highlights/01-Harness多智能体编排与分层记忆.md#8-本地语音输入与中文识别)。

### 3.5 核心代码：检测任务所有者过滤

~~~java
private void applyOwnerFilter(
        LambdaQueryWrapper<DetectionTask> wrapper) {
    Authentication authentication =
            SecurityContextHolder.getContext().getAuthentication();
    if (accessPolicy.isAdmin(authentication)) {
        return;
    }
~~~

## 四、亮点三：Kafka、OSS 与 Python Worker 异步检测

### 4.1 为什么采用异步架构

图片推理耗时长、批量任务图片多、GPU/ONNX 环境与 Java Web 环境不同。同步调用会长期占用请求线程，并把 Worker 短暂故障直接传递给用户请求。Kafka 将任务创建与推理解耦，使 Worker 可以独立扩容。

检测链路没有直接采用 RPC，原因不是 RPC 本身不可靠，而是两种通信方式适合的业务类型不同。RPC 更适合低延迟、需要立即得到结果的同步查询；图片检测包含 OSS 下载、ONNX 推理、标注图生成和结果上传，属于耗时、可排队、允许稍后返回结果的计算任务。

| 对比项 | RPC 同步调用 | Kafka 异步消息 | 本项目的选择依据 |
| --- | --- | --- | --- |
| 返回方式 | 请求必须等待 Worker 响应 | 创建任务后立即返回 taskId | 前端通过 taskId 轮询进度，无需保持长连接 |
| Worker 故障 | 超时或连接异常直接传递给 Web 请求 | 消息保留，Worker 恢复后继续消费 | 推理服务短暂不可用时任务不立即丢失 |
| 高峰流量 | 请求直接冲击 Worker | Kafka 暂存积压并按消费能力处理 | 批量上传时实现削峰和背压 |
| 扩容方式 | 依赖服务发现和负载均衡 | 同一消费者组增加 Worker | Python 推理节点可独立水平扩容 |
| 失败恢复 | 调用方自行设计超时、重试和去重 | 通过消息重投与 Offset 控制 | 配合业务幂等实现至少一次处理 |
| Java/Python 耦合 | 需要维护 RPC 地址、客户端和接口 | 双方只约定事件结构 | Spring Boot 与 Python Worker 独立部署 |

因此，Kafka 负责检测主链路的任务派发和结果回传；RPC 或 HTTP 仍可用于健康检查、配置查询等需要即时响应的操作。Kafka 并不天然保证业务只执行一次，本项目采用“至少一次投递 + dispatchId + eventId + 数据库事务”的组合保证最终状态正确。

### 4.3 detection_task 的可靠性字段

| 字段 | 作用 |
| --- | --- |
| task_id | 对外任务编号 |
| workflow_uuid | 可供智能助手和追溯接口引用的工作流编号 |
| status / stage | 技术执行状态与当前阶段 |
| dispatch_id | 当前一次派发标识，隔离重试前的旧结果 |
| last_finished_event_id | 最后处理的完成事件，防止重复更新 |
| total_images | 总图片数 |
| processed_images | 已处理图片数 |
| successful_images / failed_images | 部分失败统计 |
| source_oss_prefix | 原图对象前缀 |
| result_oss_prefix | 结果对象前缀 |
| result_json_oss_key | 结构化结果文件 |
| original_image_keys_json | 原图对象 Key 列表 |
| preview_image_keys_json | 标注图对象 Key 列表 |
| statistics_json | 聚合统计 |
| error_message | 任务失败原因 |

#### dispatchId：一次派发的批次标识

`taskId` 标识整个检测任务，而 `dispatchId` 标识该任务的某一次执行。任务超时、失败或人工重试时，`taskId` 保持不变，但系统会生成新的 `dispatchId` 并写入 `detection_task.dispatch_id`。

~~~text
TASK-1001 第一次派发：DISPATCH-A
        │
        ├── Worker A 处理较慢
        │
        └── 用户重新检测
                │
                ▼
TASK-1001 第二次派发：DISPATCH-B（数据库当前值）
        │
        ▼
Worker A 返回 DISPATCH-A 的旧结果
        │
        ▼
DISPATCH-A != DISPATCH-B → 忽略旧结果
~~~

如果不校验 `dispatchId`，第一次派发晚到的结果可能覆盖第二次检测的新结果。因此它解决的是“旧批次结果覆盖当前批次”的乱序问题。

#### eventId：一条业务事件的唯一标识

Kafka 采用至少一次投递时，同一条 `detection.task.finished` 可能因消费者重启、网络波动或 Offset 提交时机而再次到达。Worker 发布完成事件时生成 `eventId`，后端成功处理后将其保存到 `last_finished_event_id`。

~~~text
第一次收到 EVENT-9001
    → dispatchId 校验通过
    → 保存结果、更新状态
    → last_finished_event_id = EVENT-9001

Kafka 再次投递 EVENT-9001
    → 与 last_finished_event_id 相同
    → 忽略，不重复更新
~~~

`eventId` 防止重复保存结果、重复累计统计或重复触发后续动作。它解决的是“同一事件重复消费”，不能替代用于派发版本判断的 `dispatchId`。

#### 后端校验顺序

后端收到完成事件后，在事务内锁定任务，先用 `dispatchId` 拒绝过期派发，再用 `eventId` 拒绝重复事件，最后保存结果、更新任务状态并记录已处理事件。

~~~text
根据 taskId 锁定 detection_task
        │
        ▼
dispatchId 是否等于当前派发？ ──否──> 忽略旧结果
        │是
        ▼
eventId 是否已经处理？ ───────是──> 忽略重复事件
        │否
        ▼
保存结果 + 更新状态 + 记录 eventId（同一事务）
~~~

三个标识的关系是：一个 `taskId` 可以经历多个 `dispatchId`，每次派发又会产生具有独立 `eventId` 的创建事件和完成事件。

### 4.4 核心代码：Kafka 发送确认

~~~java
kafkaTemplate.send(topic, taskId, event)
        .get(sendTimeoutMs, TimeUnit.MILLISECONDS);
~~~

发送端等待 Broker 返回结果。如果超时、线程中断或发送失败，业务不会在消息尚未到达 Kafka 时错误地把任务视为已派发。

### 4.5 核心代码：Worker 先发布结果再提交 Offset

文件：kafka_detection_worker.py

~~~python
def publish_finished_and_commit(
        producer, consumer, source_message,
        topic, key, payload, timeout_seconds):
    delivery = {}
    producer.produce(
        topic=topic,
        key=key,
        value=payload,
        callback=lambda err, msg: delivery.update(error=err)
    )
    remaining = producer.flush(timeout_seconds)
    if remaining != 0 or delivery.get("error") is not None:
        raise RuntimeError("finished event delivery failed")
    consumer.commit(message=source_message)
~~~

顺序必须是：处理图片 → 上传结果 → 发布 finished → 确认发送成功 → 提交 created 消息。若先提交 Offset，再发布结果失败，Spring Boot 将永远收不到完成通知。

### 4.6 核心代码：旧事件与重复事件隔离

~~~java
boolean staleDispatch =
        StringUtils.hasText(task.getDispatchId())
        && !Objects.equals(
                task.getDispatchId(), event.getDispatchId());
boolean duplicateEvent =
        StringUtils.hasText(event.getEventId())
        && Objects.equals(
                task.getLastFinishedEventId(), event.getEventId());

if (staleDispatch || duplicateEvent) {
    return;
}
~~~

dispatchId 解决“旧任务结果覆盖新重试”的问题；eventId 解决“同一完成事件重复投递”的问题。两者不能互相替代。

### 4.7 部分失败

Worker 分别统计 successfulImages 和 failedImages：

- 全部成功：COMPLETED。
- 部分成功：PARTIAL_FAILED，保留成功图片结果。
- 全部失败或任务级异常：FAILED，并记录 errorMessage。

| avg_inference_ms | 平均推理耗时 |
| usage_count / last_used_at | 使用统计 |
| deployment_strategy | FULL、CANARY、AB_TEST、ROLLBACK |
| canary_percent / ab_group | 灰度与 A/B 配置元数据 |
| rollback_from_model_id | 回滚来源模型 |

### 6.3 核心代码：模型验证

~~~java
ModelValidationService.ValidationResult result =
        modelValidationService.validate(
                Path.of(modelInfo.getModelPath()));
modelInfo.setValidationStatus(result.status());
modelInfo.setValidationMessage(result.message());

if ("PASSED".equals(result.status())) {
    if (!"PUBLISHED".equals(modelInfo.getStatus())) {
        modelInfo.setStatus("READY");
    }
} else {
    modelInfo.setStatus("DRAFT");
    modelInfo.setIsDefault(false);
}
~~~

模型发布前必须经过 ONNX Runtime 加载校验。校验失败时模型保持 DRAFT，并取消默认标记，避免不可加载文件进入任务链路。

### 6.4 核心代码：当前模型选择逻辑

~~~java
if (modelId != null) {
    return modelInfoMapper.selectByModelId(modelId);
}
return allModels.stream()
        .filter(model -> Boolean.TRUE.equals(model.getIsDefault()))
        .findFirst()
        .orElseGet(() -> allModels.stream()
                .filter(model -> "PUBLISHED".equals(model.getStatus()))
                .findFirst()
                .orElse(allModels.get(0)));
~~~

当前任务选择顺序是：显式 modelId → 默认模型 → 第一个已发布模型 → 列表第一个模型。任务创建后会固化 modelId、modelVersion 和 threshold，保证历史结果可追溯。

### 6.5 当前限制：灰度策略尚未进入任务路由

系统已经能够保存 deploymentStrategy、canaryPercent、abGroup 和 rollbackFromModelId，也能校验配置合法性；但当前 resolveModelInfo 尚未根据这些字段做流量分配。因此正确表述是：

- 已实现模型评估、发布、默认模型、停用、归档及灰度配置管理。
- 尚未实现按灰度百分比或 A/B 分组自动选择模型。
- 后续可使用用户、设备或批次的一致性哈希完成稳定分流。

### 7.1 部署框架图

~~~mermaid
flowchart TB
~~~

### 7.2 Compose 设计

- backend-1 与 backend-2 使用同一镜像。
- 两个后端只 expose 8080，不直接暴露宿主机端口。
- Nginx 等待两个后端健康后启动。
- 上传目录挂载到同一命名卷。
- 数据库、Redis、Kafka 和 OSS 通过环境变量连接外部基础设施。

### 7.3 Nginx 负载均衡

~~~nginx
upstream doorhandle_backend {
    least_conn;
    server backend-1:8080 max_fails=3 fail_timeout=10s;
    server backend-2:8080 max_fails=3 fail_timeout=10s;
    keepalive 32;
}
~~~

least_conn 更适合耗时差异明显的请求，例如普通查询、报告生成和 Agent 长请求。新请求优先进入当前连接数较少的实例。

### 7.4 SSE 专用配置

~~~nginx
location = /api/chat-assistant/messages/stream {
    proxy_pass http://doorhandle_backend;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
    add_header X-Accel-Buffering no;
}
~~~

关闭代理缓冲后，status 和 chunk 事件可以立即到达浏览器，而不会被 Nginx 聚合后一次性返回。

### 7.5 Redis 的真实使用方式

RedisCacheManager 只有在 app.redis.enabled=true 时启用；默认 spring.cache.type 是 simple。因此：

- Redis 是可选的分布式缓存，不是聊天 Checkpoint 的主存储。
- 聊天会话与 Checkpoint 实际存储在 MySQL。
- 双实例部署时应显式启用 Redis，避免每个实例维护独立本地缓存。

### 7.6 容器安全与健康检查

- 后端采用 Maven 构建、JRE 运行的多阶段镜像。
- Java 进程使用普通 app 用户运行。
- /actuator/health 用于容器健康检查。
- 敏感参数由环境变量注入。
- Nginx 统一暴露入口并设置基本安全响应头。

### 7.7 当前限制

双实例不等于完整高可用：

- Nginx 仍是单点。
- Compose 通常部署在单台主机上。
- Kafka、MySQL、Redis 的高可用由外部基础设施负责。
- 当前没有配置中心、服务注册发现和分布式链路追踪。

## 十、两分钟项目介绍

这是一个面向工业门把手质检的视觉检测与质量追溯平台。前端使用 Vue，后端使用 Spring Boot 和 MySQL，图片存储在 OSS，检测任务通过 Kafka 异步派发给 Python ONNX Worker。

系统的第一个亮点是 Harness Agent。它不是简单的大模型问答，而是使用 StateGraph 编排路由、槽位补全、检测、资源、报表和运维智能体。高风险修改操作会先生成待确认动作，把状态保存到 MySQL Checkpoint，用户确认后再恢复执行。系统还结合 Mem0 用户记忆和 ChromaDB RAG，分别处理个性化历史和公共业务知识。

第二个亮点是异步检测可靠性。前端通过预签名 URL 把图片直接上传 OSS，Spring Boot 生成 dispatchId 并发布 Kafka 任务。Python Worker 推理完成后先发布 finished 事件，确认成功后才提交原消息 Offset。后端再通过 dispatchId 防止旧结果覆盖，通过 eventId 防止重复事件更新。

第三个亮点是质量闭环。检测结果会进入分派、人工复核、缺陷处置、返工和复检流程。数据库保存模型版本、阈值、缺陷证据、误报数、操作人和阶段时间，因此可以按任务、批次或工单回溯完整过程。

部署层通过 Nginx 和 Docker Compose 运行双后端实例，对普通 API 使用 least_conn 负载均衡，对 Agent SSE 单独关闭缓冲并延长超时。当前系统已经具备用户级逻辑隔离，但还不是完整的企业级 tenant_id 多租户；模型灰度配置也尚未进入自动选型链路。这些限制会作为后续微服务化和分布式演进的重点。

## 十一、面试中必须说清楚的边界

- 可以说：实现了用户级逻辑隔离。
- 不应说：已经实现企业级 SaaS 多租户。
- 可以说：模型支持灰度和 A/B 配置管理。
- 不应说：任务已经按灰度比例自动路由。
- 可以说：Compose 部署双后端实例并支持故障摘除。
- 不应说：单机 Compose 已经实现跨主机高可用。
- 可以说：Kafka 链路采用至少一次语义并做业务幂等。
- 不应说：Kafka 天然保证端到端只执行一次。
